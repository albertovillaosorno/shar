# SHA-256

This non-governing record documents a cryptographic hash function used for
integrity and identity evidence without treating a digest as proof of ownership,
authenticity, authorization, or legal compliance.

- Counsel review: Not performed.

## Covered Material

SHA-256 digests used by SHAR for source snapshots, manifests, converted
artifacts, package evidence, and deterministic comparison.

## Repository Use And Scope

SHAR computes SHA-256 values over bounded byte sequences to identify exact file
content and detect change. A digest records a mathematical relationship to
bytes; it does not identify who created those bytes or whether possession,
modification, or distribution is authorized.

## Provenance And Version History

SHA-256 is part of the NIST Secure Hash Standard. Standards revisions,
implementation libraries, hardware acceleration, and serialization conventions
must be distinguished from the algorithm identity itself.

## Authorship, Ownership, And Attribution

NIST and standards contributors retain applicable rights in standards
publications. Implementations retain separate authorship and licensing. Rights
in hashed content remain unchanged.

## License Or Terms Basis

Use of the standardized algorithm does not license the hashed data or a specific
software implementation. Any bundled implementation and its dependencies require
separate license and notice review.

## Distribution, Modification, And Compatibility

A matching digest supports byte-for-byte identity under the stated algorithm and
encoding. It does not prove lawful acquisition, provenance, malware safety,
meaning, or absence of collision risk in every adversarial context.

## Compliance Posture

- Record the algorithm name, byte scope, encoding, and digest representation.
- Never truncate a digest without documenting the security and identity impact.
- Keep private sample hashes private when publication would reveal sensitive
  inventory information.
- Do not use a digest as a substitute for signatures or trusted provenance.
- Recheck current cryptographic guidance for security-sensitive uses.

## Technical Baseline And SHAR Profile

### Public baseline

FIPS 180-4 defines SHA-256 as a secure hash algorithm producing a 256-bit
message digest. The algorithm definition does not specify which application
bytes must be hashed, whether data is canonicalized first, how the digest is
rendered, whether it may be truncated, or whether publishing it is safe.

### SHAR profile

The RMV domain contains an independently authored SHA-256 implementation with a
narrow deterministic profile:

- the input is the exact caller-provided byte slice;
- the output is the complete 32-byte digest;
- textual rendering is exactly 64 lowercase hexadecimal characters;
- no prefix, separators, uppercase conversion, or truncation is used; and
- regression tests include the standard empty-message and `abc` vectors.

The digest is used as evidence and identity for movie inputs and candidates. A
hash is not anonymization: it may still disclose equality, permit dictionary
matching, or identify known proprietary content.

### Acceptance boundary

Every manifest field using SHA-256 must identify the exact byte scope, any
pre-hash serialization or normalization, the displayed representation, privacy
classification, collision handling, and whether the digest may be published.
Different artifact families must not silently reuse a digest label for different
byte scopes.
