# 17 U.S.C. § 102(b) — Ideas, Processes, Systems, And Methods

## Question Presented

Which functional elements of a file format, interface, command hierarchy, or
computer program fall outside copyright protection, and which expressive
elements may remain protected?

## Verified Statutory Baseline

Section 102(b) states that copyright protection does not extend to an idea,
procedure, process, system, method of operation, concept, principle, or
discovery, regardless of the form in which it is described, explained,
illustrated, or embodied.

The current official legislative notes explain that computer programs are
protected to the extent that they contain the programmer's expression of
original ideas, while the actual processes or methods embodied in the program
are outside copyright's scope. The statute preserves the idea-expression
boundary; it does not classify an entire software artifact or format by label.

## Required Element-By-Element Analysis

For each asserted SHAR element, identify:

1. The exact bytes, fields, command labels, hierarchy, ordering, or behavior.
1. Whether compatibility, hardware, efficiency, or a public standard requires
   that element.
1. Which expressive alternatives remain technically viable.
1. Whether SHAR copied expression or independently inferred functionality.
1. Whether original selection, coordination, arrangement, documentation, or
   implementation code remains at issue.
1. The controlling jurisdiction and later judicial authority.

## Not Established

- That every file format or interface is categorically uncopyrightable.
- That every command label or hierarchy is a method of operation.
- That expressive documentation, examples, code, selection, or arrangement is
  free to copy.
- That independent implementation resolves contract, patent, trade-secret,
  trademark, or anti-circumvention questions.
- That the statute authorizes acquisition or publication of protected material.

## Repository Posture

- Describe functional requirements without reproducing unnecessary upstream
  expression.
- Use repository-authored names, organization, examples, and documentation where
  compatibility does not require an upstream element.
- Preserve evidence showing why a compatible element is technically necessary.
- Avoid format-wide or interface-wide legal conclusions.
